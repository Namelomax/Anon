# Математика и устройство анонимайзера

Документ объясняет, **как работает приложение по шагам** и **что означают метрики**
`P, R, F1, gold, cover%, miss`, с математической точки зрения.

---

## Часть 1. Как работает приложение (по этапам)

Вход — текст `T` (строка символов). Цель — заменить персональные данные на
плейсхолдеры и сохранить обратимый словарь (mapping).

```
T ──▶ [1] Детекторы ──▶ спаны ──▶ [2] Разрешение пересечений
   ──▶ [3] Присвоение плейсхолдеров ──▶ [4] Замена всех вхождений
   ──▶ (обезличенный текст T', mapping M)
                         │
         деанонимизация: T = replace(T', M⁻¹)
```

### Базовое понятие — спан (span)

Спан — это найденный фрагмент:
```
span = (start, end, label, text)
```
где `start, end` — позиции символов в `T` (полуинтервал `[start, end)`),
`label` — тип (PERSON, PHONE, INN…), `text = T[start:end]`.

Два спана `a` и `b` **пересекаются**, если их интервалы перекрываются:
```
overlap(a, b)  ⟺  a.start < b.end  И  b.start < a.end
```

### Этап 1. Детекторы (три слоя)

Каждый слой — это функция `find(T) → список спанов`. Слои независимы, их выходы
объединяются.

**(а) Регулярные выражения** — детерминированные. Для текста PII с жёстким
форматом (телефон, email, ИНН, СНИЛС, паспорт…). Математически — конечный
автомат, который либо совпал с шаблоном, либо нет. Вероятностей нет:
score ∈ {0, 1}.

**(б) NER-модель (GLiNER / Natasha)** — вероятностная. Для «размытого» (ФИО,
адреса, организации). Модель для каждого кандидата-спана и каждой метки выдаёт
**оценку уверенности** `s ∈ [0, 1]` (через сигмоиду):
```
s = σ(z) = 1 / (1 + e^(−z))
```
где `z` — «логит» (необработанный выход нейросети). Спан принимается, если
`s ≥ τ` (порог). Подробнее про `τ` — в Части 3.

> GLiNER ограничен длиной контекста (~512 токенов). Поэтому длинные документы
> **разбиваются на куски** (`_chunks`), модель применяется к каждому, а позиции
> спанов сдвигаются обратно: `start_глоб = offset_куска + start_локал`.

**(в) Локальная LLM** — добивает то, что не поймали (числа прописью, латиница,
склонения). Модель возвращает подстроки, которые мы **сами находим** в тексте
(`str.find`) и превращаем в спаны. Что не нашли дословно — отбрасываем.

**Фильтр не-PII.** Спаны PERSON/ORG, являющиеся должностями («Главный бухгалтер»)
или софтом («1С:Бухгалтерия»), удаляются — это не персональные данные.

### Этап 2. Разрешение пересечений (overlap resolution)

Разные слои находят пересекающиеся спаны. Нужно оставить непересекающийся набор.
Жадный алгоритм: сортируем все спаны по «силе»
```
rank(span) = ( priority[label], length, −start )    (по убыванию)
```
и идём сверху вниз, добавляя спан, только если он **не пересекается** с уже
принятыми. `priority` — заданные веса (например, EMAIL=90 важнее PHONE=40), чтобы
конкретный тип «победил» широкий. Результат — спаны без пересечений, отсортированы
по `start`.

### Этап 3. Присвоение плейсхолдеров

Группируем спаны по **нормализованному значению** (`label` + схлопнутые пробелы +
регистр). Каждой уникальной сущности — один плейсхолдер `[LABEL_N]`, нумерация по
порядку первого появления. Получаем биекцию (взаимно-однозначное соответствие):
```
M : плейсхолдер → оригинал        (mapping)
```
Одинаковое значение → один плейсхолдер (например, один человек везде `[PERSON_1]`).

### Этап 4. Замена всех вхождений

Как только значение признано чувствительным, заменяем **каждое** его точное
вхождение в `T` (длинные значения — раньше, чтобы вложенные не ломали замену). Это
страхует от пропусков, когда NER нашёл сущность в одном месте, но пропустил повтор.

### Деанонимизация (обратный модуль, без ИИ)

Чисто детерминированная замена: по `M` строим один регэксп из всех плейсхолдеров и
заменяем каждый на оригинал за один проход. Поэтому
```
deanonymize(anonymize(T)) = T      (точное восстановление)
```
если ни один оригинал не содержит плейсхолдер другого (на практике — да).

## Часть 2. Метрики: P, R, F1, gold, cover%, miss

### Матрица ошибок (confusion matrix)

Сравниваем предсказанные спаны с эталонными («gold», ручная разметка бенчмарка).
Для каждой категории (типа) считаем:

| Обозначение | Что это | Кратко |
|---|---|---|
| **TP** (true positive) | предсказали, и это правда есть в эталоне | «верное попадание» |
| **FP** (false positive) | предсказали, но в эталоне нет | «лишнее» |
| **FN** (false negative) | в эталоне есть, но не предсказали | «пропуск» |

**Что значит «совпало»?** Здесь используется **overlap matching** (совпадение по
пересечению, как в бенчмарке redmadrobot): предсказание засчитывается, если оно
**пересекается** с эталонным спаном **той же категории**. Перед этим смежные спаны
одной категории сливаются (`merge_adjacent`). Это мягче, чем «точное совпадение
границ», и устойчивее к мелким сдвигам.

### Precision (точность) — P

> Из того, что мы пометили, какая доля верна?

```
P = TP / (TP + FP) = TP / (всего предсказано)
```
Низкая `P` ⇒ много «лишнего» (над-редактирование).

### Recall (полнота) — R

> Из того, что реально есть, какую долю мы нашли?

```
R = TP / (TP + FN) = TP / (всего в эталоне)
```
Низкая `R` ⇒ много пропусков (для анонимизации это утечки — самое опасное).

### F1 — гармоническое среднее P и R

```
F1 = 2 · P · R / (P + R)
```
Гармоническое (а не обычное) среднее **штрафует перекос**: если `P=1.0`, а `R=0.1`,
то F1 ≈ 0.18, а не 0.55. То есть нельзя «выехать» только на одной метрике.

### Микро- vs макро-усреднение

В отчёте — **micro-F1** по строке `ALL (micro)`: суммируем TP/FP/FN по всем
категориям, потом считаем P/R/F1. Это даёт вес пропорционально количеству спанов
(частые типы влияют сильнее). Альтернатива — macro (среднее F1 по категориям, все
типы равноправны). Лидерборд и мы используем micro.

### Колонки `gold`, `cover%`, `miss` (метрики приватности)

Это **наше дополнение** к стандартным P/R/F1, заточенное под анонимизацию, где
важно «замаскировано вообще или нет», **а не** правильность метки.

- **`gold`** — сколько эталонных спанов этого типа всего (знаменатель).

- **`cover%`** — доля эталонных спанов, перекрытых **хоть каким** предсказанием
  (любой категории). Это «privacy-recall»: замаскировано ли место в принципе?
  ```
  cover% = (gold-спаны, перекрытые любым предсказанием) / gold
  ```
  Отличие от обычного `R`: в `R` нужна **та же** метка, в `cover%` — **любая**.
  Поэтому СНИЛС, замаскированный как `[INN_1]`, даёт `cover%`, но не `R`.

- **`miss`** — сколько эталонных спанов **не перекрыто ничем** (реальные утечки):
  ```
  miss = gold − (перекрытые любым предсказанием)
  ```
  **Это главный показатель для анонимизации.** Цель — `miss → 0`.

Итоговая строка `КЛЮЧЕВОЕ: замаскировано X/Y, ПРОПУЩЕНО СОВСЕМ: miss` агрегирует это
по всему набору.

### Три исхода для каждого эталонного спана (в разборе ошибок)

```
если перекрыт предсказанием той же категории   → TP        (всё верно)
иначе если перекрыт предсказанием др. категории → MISMATCH  («не та категория», но скрыто)
иначе                                           → TRUEMISS  («пропущено совсем» = утечка)
```
Для приватности `MISMATCH` почти безвреден, `TRUEMISS` — опасен.

## Часть 3. Порог, Precision–Recall и ROC

### Порог τ у NER

GLiNER принимает спан при `score ≥ τ`. Меняя `τ`, мы двигаемся по компромиссу:

```
τ ↓ (ниже)  ⇒  принимаем больше спанов  ⇒  R ↑, P ↓   (больше находим, но больше лишнего)
τ ↑ (выше)  ⇒  принимаем меньше         ⇒  P ↑, R ↓   (точнее, но больше пропусков)
```

Для нашей задачи (минимум утечек) ставим `τ` **низким** (0.3): жертвуем точностью
ради полноты, потому что «лишняя маскировка» не страшна, а пропуск = утечка.

### Precision–Recall кривая

Если перебрать все `τ ∈ [0, 1]` и для каждого посчитать `(R, P)`, получится
**PR-кривая**. Каждая точка — один режим работы. «Best F1 threshold» из лидерборда —
это `τ`, дающий максимум `F1` на этой кривой. Площадь под PR-кривой (AUC-PR) —
сводная мера качества, не зависящая от выбора `τ`.

### ROC-кривая (то, о чём ты слышал)

ROC строит **TPR vs FPR** при разных `τ`:
```
TPR = R = TP / (TP + FN)            (по оси Y)
FPR = FP / (FP + TN)                (по оси X)
```
Проблема для NER/PII: **TN (true negative) плохо определён** — «сколько мест мы
верно НЕ пометили» зависит от того, что считать «местом» (каждый символ? токен?
возможный спан?). Поэтому в задачах поиска сущностей/PII обычно используют
**PR-кривую, а не ROC** — она не требует `TN`. В этом проекте мы ROC не считаем;
аналог «движения по кривой» — это изменение порога GLiNER (`--gliner-threshold`).

Кратко: **ROC** хорош для сбалансированной бинарной классификации; **PR** —
для редких событий и поиска (наш случай).

## Часть 4. Числовой пример

Текст: `«Иванов работает в Москве. Иванову позвонили на +7 900 123-45-67.»`

Эталон (gold): `Иванов`=PERSON, `Иванову`=PERSON, `Москве`=LOCATION,
`+7 900 123-45-67`=PHONE → всего gold: PERSON=2, LOCATION=1, PHONE=1.

Предсказания: регэксп нашёл телефон; GLiNER нашёл `Иванов`=PERSON и `Москве`=LOCATION,
но пропустил склонённое `Иванову`. Замена всех вхождений добавит `Иванову` (как
повтор `Иванов`? нет — это другая словоформа, точного совпадения нет → останется).

Считаем PERSON:
```
gold = 2 (Иванов, Иванову)
TP = 1 (Иванов — совпал)
FN = 1 (Иванову — пропуск)
FP = 0
P = 1/(1+0) = 1.000
R = 1/(1+1) = 0.500
F1 = 2·1·0.5/(1+0.5) = 0.667
cover% = 1/2 = 0.500   (Иванову не перекрыт ничем)
miss = 2 − 1 = 1       ← одна утечка
```
LOCATION и PHONE: `P=R=F1=1.0`, `cover%=1.0`, `miss=0`.

ALL (micro): `TP=3, FP=0, FN=1` →
```
P = 3/3 = 1.000,  R = 3/4 = 0.750,  F1 = 0.857
замаскировано = 3/4 = 75%,  утечек = 1
```

Вывод по примеру: метка везде верная (`P=1`), но одно склонённое имя утекло —
именно для таких случаев нужен LLM-слой или морфология.

## Часть 5. Какие метки ищет каждый слой

**Regex — основной набор (всегда), 13 меток:**
`EMAIL`, `URL`, `IP_ADDRESS`, `PHONE`, `CREDIT_CARD`, `INN`, `SNILS`, `OMS`,
`PASSPORT`, `DRIVER_LICENSE`, `MILITARY_ID`, `BIRTH_CERTIFICATE`,
`PERSON` (только формат «Фамилия И.О.»).

**Regex — корпоративный слой (`--corporate`), 5 меток:**
`AMOUNT`, `CONTRACT`, `DATE`, `ORG` (ООО/АО/ГК «…»), `FILE`.

**NER:**
- GLiNER — промпты `person / location / organization / address` → `PERSON`, `LOCATION`, `ORG`.
- Natasha — `PER→PERSON`, `LOC→LOCATION` (+`ORG`, если `include_org`).

**LLM, 14 меток:** `PERSON`, `LOCATION`, `PHONE`, `EMAIL`, `URL`, `IP_ADDRESS`,
`CREDIT_CARD`, `INN`, `SNILS`, `OMS`, `PASSPORT`, `DRIVER_LICENSE`, `MILITARY_ID`,
`BIRTH_CERTIFICATE`. Свёртка `FIRST/LAST/MIDDLE_NAME→PERSON`,
`CITY/STREET/HOUSE/REGION/COUNTRY/DISTRICT→LOCATION`; `ORGANIZATION`/`DATE`
отфильтровываются whitelist'ом.

| Метка | Regex | NER | LLM |
|---|:---:|:---:|:---:|
| EMAIL, URL, IP_ADDRESS, PHONE, CREDIT_CARD | ✅ | — | ✅ |
| INN, SNILS, OMS, PASSPORT, DRIVER_LICENSE, MILITARY_ID, BIRTH_CERTIFICATE | ✅ | — | ✅ |
| PERSON | ✅ (только И.О.) | ✅ | ✅ |
| LOCATION | — | ✅ | ✅ |
| ORG | ✅ (`--corporate`) | ✅ | — |
| AMOUNT, CONTRACT, DATE, FILE | ✅ (`--corporate`) | — | — |

Разделение труда: **regex** — форматные данные (документы/контакты/корпоративное),
**NER** — размытые ФИО/адреса/организации, **LLM** — нестандартное (склонения,
латиница, числа прописью).

## Краткая шпаргалка

| Метрика | Формула | Смысл | Хотим |
|---|---|---|---|
| P | TP/(TP+FP) | доля верных среди предсказанных | ↑ (меньше лишнего) |
| R | TP/(TP+FN) | доля найденных среди реальных | ↑ (меньше пропусков) |
| F1 | 2PR/(P+R) | баланс P и R | ↑ |
| cover% | перекрыто_любым/gold | замаскировано хоть как-то | ↑ (приватность) |
| miss | gold − перекрыто_любым | реальные утечки | **→ 0** |

Для анонимизации **главное — `miss` (→ 0) и `cover%` (→ 100%)**; `P` вторично
(над-редактирование терпимо), `F1` — общий ориентир качества.
